# SelfApp 产品需求文档（PRD）— 功能完善版

> **版本**: v1.0 | **状态**: 待开发 | **面向对象**: 技术开发团队  
> **产品定位**: 个人使用的 Android 自律工具 — 选好要限制的 App，到点自动锁死，想玩也玩不了。

---

## 1. 功能概述

### 1.1 业务目标

帮助用户建立**数字自律习惯**，通过技术手段强制限制手机使用，解决「明知不该刷但停不下来」的问题。核心价值不是「阻止」，而是**在用户意志力薄弱时提供外部约束**。

### 1.2 目标用户画像

| 维度 | 描述 |
|------|------|
| **核心用户** | 18-35 岁，学生/职场新人，有明确自律需求 |
| **使用场景** | 深夜刷短视频失控、学习/工作时忍不住看手机、作息不规律 |
| **心理特征** | 意志力有波动性 — 理智时想约束自己，冲动时想放弃约束 |
| **技术能力** | 普通智能手机用户，能完成基本权限授权 |

### 1.3 与现有系统关系

本应用基于已有的 Flutter + Android 原生架构（详见 `PROJECT.md`），本文档聚焦于**功能缺口补全**和**体验优化**，不重复已有架构设计。

## 2. 需要完善的功能清单（按优先级排序）

### P0 — 决定产品可用性的核心缺陷修复

#### 2.1 手动守护时长可配置

**现状问题**：`home_page.dart:155` 中硬编码 `startManual(25)`，用户无法选择守护时长。

**功能设计**：

```
用户点击手动守护开关时：
├── 若已选应用数 = 0 → 显示 Toast "请先选择至少一个应用"，不弹窗
├── 若已选应用数 > 0 → 弹出时长选择底部面板（BottomSheet）
│   ├── 标题："选择守护时长"
│   ├── 6 个预设选项（网格布局，2列3行）：
│   │   ├── [15 分钟]  — 适合短时专注
│   │   ├── [25 分钟]  — 番茄钟（默认选中）
│   │   ├── [45 分钟]  — 一节课长度
│   │   ├── [1 小时]   — 深度工作
│   │   ├── [2 小时]   — 长时段专注
│   │   └── [自定义...] — 点击后弹出时间输入器
│   └── 底部操作栏：
│       ├── 左侧：[取消] 文字按钮
│       └── 右侧：[开始守护] 主按钮（primary 色，圆角 pill）
│
└── 用户点击「开始守护」→ 关闭弹窗 → 调用 startManual(选定分钟数) → 跳转锁屏页
```

**交互细节**：

| 项目 | 规则 |
|------|------|
| 默认选项 | 25 分钟（番茄钟，符合认知习惯） |
| 自定义范围 | 5 分钟 ~ 480 分钟（8小时），步进 5 分钟 |
| 上次记忆 | 记住用户上次选择的时长，下次打开自动选中 |
| 弹窗动画 | 从底部滑入，背景模糊遮罩（alpha 0.4） |
| 取消行为 | 关闭弹窗，Switch 不改变状态 |

**数据存储**：新增 DataStore key `preferred_guard_duration_minutes`（Int，默认 25）

#### 2.2 困难退出机制（锁屏页退出按钮）

**现状问题**：锁屏页 `PopScope(canPop: false)` 完全禁止退出。定时模式下用户被永久锁定，只能强制杀进程——这在实际使用中会造成恐慌和不信任。

**设计理念**：**退出应该「可能但困难」**。这不是为了折磨用户，而是利用「摩擦力」让冲动退出的成本足够高，给理智回归争取时间。每次点击等待期间展示励志文案，将「退出尝试」转化为「自我对话」的机会。

**交互规范**：

| 行为 | 处理方式 |
|------|----------|
| **倒计时期间按钮点击** | 按钮显示剩余秒数（如 `继续退出 (12s)`），点击无反应 + 轻微震动反馈 |
| **倒计时归零** | 按钮变为可点击态（颜色变亮/脉冲动画），文字恢复原始文案 |
| **用户按返回键/Home 键** | 退出确认弹窗不关闭（覆盖在锁屏之上），AccessibilityService 继续拦截 |
| **等待期间屏幕熄灭/应用切后台** | 倒计时继续运行（基于 SystemClock.elapsedRealtime，非 wall time）；回到前台时刷新显示 |
| **等待超时无操作** | 不自动关闭弹窗，保持当前步骤状态不变；用户随时可以继续 |
| **中途取消（点击弹窗外部区域）** | **不允许** — 弹窗无遮罩层点击关闭，唯一途径是走完流程或点击左上角「返回守护」（重置到步骤 0） |
| **全部 5 步完成后** | 调用 `stopGuard()` → 显示「守护已结束」总结页 → 3 秒后自动跳转首页或点击「返回首页」立即跳转 |

#### 2.3 启动守护二次确认

**现状问题**：Switch 开启后立即调用 `startManual()` 并 `context.push('/lock')`，用户没有任何反悔机会，误触即锁定。

**功能设计**：

```
用户打开 Switch 时：
├── Switch 先不改变状态（保持 OFF）
├── 弹出轻量确认对话框（AlertDialog，非全屏）：
│   ├── 标题："开始守护？"
│   ├── 内容："选中的 {N} 个应用将被锁定\n守护时长：{X} 分钟"
│   ├── 下方显示已选应用图标预览（最多显示 4 个 + "+N"）
│   ├── [取消] → 关闭对话框，Switch 保持 OFF
│   └── [开始守护] → 关闭对话框 → Switch 变 ON → 执行启动逻辑
│
├── 若用户 5 秒内未操作 → 自动视为取消（对话框消失）
└── 对话框出现时播放轻微震动（HapticFeedback.mediumImpact）
```

#### 2.5 应用图标真实加载

**现状问题**：`app_select_page.dart:164` 所有应用显示同一个灰色占位图标 `Icons.apps`，用户无法通过图标辨识应用。

**功能设计**：

**前提条件**：NativeChannel.getInstalledApps() 需要额外返回每个应用的图标数据。

**Android 端修改（NativeChannel.kt）**：

```kotlin
// 在 getInstalledApps() 的返回 map 中增加 icon 字段
// 方案：将 Drawable 转为 Base64 编码的 PNG 字符串
val icon = packageManager.getApplicationIcon(info.packageName)
val bitmap = (icon as BitmapDrawable).bitmap
val stream = ByteArrayOutputStream()
bitmap.compress(Bitmap.CompressFormat.PNG, 80, stream)
val iconBase64 = Base64.encodeToString(stream.toByteArray(), Base64.NO_WRAP)

mapOf(
    "packageName" to info.packageName,
    "label" to info.label.toString(),
    "icon" to iconBase64  // 新增
)
```

**Flutter 端渲染**：

```dart
// _AppListItem 中替换占位图标
ClipRRect(
  borderRadius: BorderRadius.circular(10),
  child: app.iconBase64 != null
    ? Image.memory(
        base64Decode(app.iconBase64!),
        width: 40,
```

### P2 — 提升产品成熟度的增强功能

#### 4.1 重启恢复机制

**现状问题**：手机重启后，前台服务和 AlarmManager 注册全部丢失，守护状态不会恢复。

**功能设计**：

```
开机流程（BootReceiver.onReceive）：
├── 1. 延迟 10 秒等待系统稳定
├── 2. 读取本地存储的守护状态（guard_active, guard_trigger_type 等）
├── 3. 若 guard_active = true：
│   ├── 若 trigger_type = MANUAL：
│   │   ├── 计算 remaining = endAt - now
│   │   ├── 若 remaining > 0：
│   │   │   ├── 重新 startForeground()
│   │   │   ├── 重新注册结束 Alarm（remaining ms 后）
│   │   │   └── 直接拉起 LockActivity
│   │   └── 若 remaining <= 0：
│   │       └── 清理守护状态（已过期）
│   │
│   └── 若 trigger_type = SCHEDULED：
│       ├── 读取关联的 schedule 配置
│       ├── 检查当前时间是否在计划时间段内：
│       │   ├── 在时间段内 → 重新启动守护 + LockActivity
│       │   └── 不在 → 仅重新注册 Alarm（等待下一个触发时间）
│       └── 重新注册所有 enabled schedule 的 Alarm
│
├── 4. 若 guard_active = false：
│   └── 仅重新注册所有 enabled schedule 的 Alarm（等待触发）
│
└── 5. 发送通知：「自律：已恢复运行」
```

**AndroidManifest 注册**：

```xml
<receiver android:name=".BootReceiver"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
    </intent-filter>
</receiver>
```

#### 4.3 触觉反馈（Haptic Feedback）

**现状问题**：全局无任何震动反馈，操作缺乏物理感知。

**添加位置**：

| 操作 | 反馈类型 | 说明 |
|------|----------|------|
| Switch 开关切换 | `mediumImpact` | 明显但不强烈 |
| 退出守护按钮点击 | `heavyImpact` | 强烈警示 |
| 退出流程每步按钮激活 | `selectionClick` | 轻微确认感 |
| 计划卡片长按选中 | `heavyImpact` | 进入编辑模式 |
| 时间选择器滚动停止 | `selectionTick` | 数字到位的确认感 |
| 星期选择切换 | `selectionClick` | 轻量确认 |
| 保存成功 | `success` | 成功反馈（如设备支持） |

**统一封装**：

```dart
class Haptic {
  static void light() => HapticFeedback.lightImpact();
  static void medium() => HapticFeedback.mediumImpact();
  static void heavy() => HapticFeedback.heavyImpact();
  static void select() => HapticFeedback.selectionClick();
  static void tick() => HapticFeedback.selectionTick();
  static void success() {
    // Android 31+ 支持 success/failure
    try { HapticFeedback.performHapticFeedback(HapticFeedbackType.confirmation); }
    catch (_) { medium(); } // 降级
  }
}
```

## 5. 边界情况处理

### 5.1 并发与时序

| 场景 | 处理方式 |
|------|----------|
| 定时守护进行中，用户手动开启守护 | 提示「定时守护正在进行中，请先结束定时守护或在定时结束后手动启动」 |
| 多个定时计划时间重叠 | 允许重叠，取**最长结束时间**作为实际结束时间；统计分别记录 |
| 守护期间手机时间被手动修改 | 基于 `SystemClock.elapsedRealtime`（单调时钟）计算倒计时，不受系统时间调整影响 |
| 快速连续点击「退出守护」按钮 | 防抖：500ms 内只响应第一次点击 |
| 应用被系统强行杀死（低内存） | BootReceiver + WorkManager 双保险恢复 |

### 5.2 权限边界

| 场景 | 处理方式 |
|------|----------|
| 用户在设置中关闭无障碍服务 | 5 秒轮询检测 → 显示断开提示 → 锁屏功能暂停（不崩溃） |
| 用户撤销悬浮窗权限 | 锁屏页无法覆盖 → 降级为普通 Activity（可被 Home 键退出）+ 提示 |
| 用户在应用详情中「强制停止」 | 所有 Service/Receiver 终止；下次启动时走初始化流程 |
| Android 系统升级后精确闹钟权限被收回 | 检测到后引导用户重新授权 |

### 5.3 存储边界

| 场景 | 处理方式 |
|------|----------|
| 存储空间不足 | 写入前检查可用空间；不足时丢弃最旧的 guard_record（FIFO） |
| 数据文件损坏 | 启动时校验 JSON 格式；损坏则用空数据初始化（不崩溃） |
| 首次安装（无历史数据） | 所有 Provider 以空初始值启动；首页显示引导提示 |

## 6. 非功能性需求

### 6.1 性能要求

| 指标 | 要求 |
|------|------|
| 首页冷启动时间 | < 1.5 秒（从点击图标到页面可交互） |
| 锁屏页打开延迟 | < 300ms（从检测到拦截到锁屏覆盖） |
| 前台应用检测间隔 | 1 秒（平衡实时性与耗电量） |
| 内存占用（前台） | < 150 MB |
| 内存占用（后台服务） | < 50 MB |
| 电池消耗（后台） | < 2%/小时（仅 ForegroundService + AccessibilityService） |

### 6.2 兼容性要求

| 维度 | 要求 |
|------|------|
| 最低 Android 版本 | API 26 (Android 8.0) |
| 目标 Android 版本 | API 35 (Android 15) |
| 屏幕尺寸适配 | 4.7" ~ 7.2"（手机 + 折叠屏） |
| 横屏支持 | 锁屏页强制竖屏；其余页面允许横屏 |
| 无障碍支持 | 应用自身 UI 需满足基本无障碍（semantics label） |
| 多语言 | 中文简体为主（暂不需国际化） |

### 6.3 安全要求

| 维度 | 要求 |
|------|------|
| 数据存储 | 全部本地存储，不上传任何数据 |
| 权限最小化 | 仅申请功能必需的权限 |
| 导出数据加密 | 备份 JSON 可选密码加密（AES-256） |
| 退出绕过防护 | 检测常见绕过手段（ADB、USB 调试等），记录异常行为 |

### 6.4 可维护性要求

| 维度 | 要求 |
|------|------|
| 代码注释 | 关键业务逻辑必须有中文注释 |
| 日志规范 | 统一日志 Tag 前缀 `SelfApp:`，区分 DEBUG/WARN/ERROR |
| 错误上报 | Release 版本捕获未处理异常，写入本地日志文件 |
| 配置外置 | 魔法数字（时长、次数、等待时间）提取为可配置常量 |
